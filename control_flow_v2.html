0. 架構總覽與 Client/BaaS 分層

本系統為典型 BaaS 架構：前端 App 處理 UI、感測器、邊緣運算；後端由 Firebase 或 Supabase 提供驗證、資料庫、雲端函數。

0.1 推薦技術棧

Client 端

  • 框架：React Native（跨平台）或原生 Swift + Kotlin
  • 感測器：CoreMotion (iOS) / SensorManager (Android)
  • NFC：CoreNFC (iOS 11+) / NfcAdapter (Android)
  • Edge VAD：TensorFlow Lite + WebRTC VAD
  • Edge Whisper：whisper.cpp tiny model（選配，Layer 3 才用）
  • 背景去背：iOS 16+ Vision API / Android ML Kit Segmentation
  • 本地儲存：SQLite（事件 queue）

BaaS 端

  • 主選擇：Firebase（Auth + Firestore + Functions + Storage）
  • 次選擇：Supabase（Postgres 更好、Realtime 更透明）
  • ML：Cloud Functions 呼叫 Anthropic / OpenAI API
  • Leaderboard：Firebase Extensions 或 Redis sorted set
  • 推播：FCM（跨平台）
  • 監控：Crashlytics + Performance Monitoring

0.2 Session 狀態機（延續 v1）

狀態：CREATED → LOBBY → ACTIVE → ENDED → ARCHIVED → CLOSED，任一階段可 ABORTED。非法轉移一律由 Cloud Functions 拒絕並記錄。

1.2 頭像去背流程

使用者上傳頭像時，系統自動產生兩個版本：原始頭像（大頭貼用）與去背版本（群組寵物頭部用）。去背使用裝置端 ML（iOS Vision / Android ML Kit Segmentation），失敗時 fallback 到 Remove.bg API。

async function uploadAvatar(file: File): Promise<{ avatar: string, cutout: string }> {
  const originalUrl = await storage.upload(file, `users/${uid}/avatar.jpg`);

  try {
    const cutoutBlob = await onDeviceSegmentation(file);  // Vision / MLKit
    const cutoutUrl = await storage.upload(cutoutBlob, `users/${uid}/cutout.png`);
    return { avatar: originalUrl, cutout: cutoutUrl };
  } catch (e) {
    const cutoutBlob = await callRemoveBgAPI(file);
    const cutoutUrl = await storage.upload(cutoutBlob, `users/${uid}/cutout.png`);
    return { avatar: originalUrl, cutout: cutoutUrl };
  }
}

2.3 群組寵物初始化 v2.1 NEW

群組建立時，成員們投票或指定一位成員成為「寵物」。被選中者的去背臉部照片作為寵物頭部，搭配成員共同選擇的去頭 emoji 動物身體。寵物在聚會期間不會出現，僅在聚會結束時依據品質評分獲得能量或損失生命值。

// 群組建立時
interface Group {
  group_id: UUID;
  name: string;
  creator_uid: UUID;
  member_uids: UUID[];
  created_at: ISO8601;

  // v2.1 新增：群組寵物系統
  pet_target_uid: UUID;                  // 被選為寵物的那個人
  pet_body_emoji: string;                // 去頭 emoji（"🐰"/"🐻"/"🐱"/"🐶"/"🦊"/"🐸"/"🐧"等）
  pet_name: string;                      // 寵物暱稱（可自訂）
  pet_energy: number;                    // 能量值 0-100（>=0 存活，=0 瀕死）
  pet_max_energy: number;                // 目前上限，隨等級成長
  pet_hp: number;                        // 生命值 0-5（歸零 = 需要急救復活）
  pet_level: number;                     // 等級（能量累積解鎖）
  pet_accumulated_score: number;         // 歷史總能量累積
  pet_accessories: string[];             // 解鎖的配件
  pet_last_fed_at: ISO8601;              // 最後一次餵食（聚會結束時）
  pet_status: "HAPPY" | "NORMAL" | "HUNGRY" | "CRITICAL";  // 依能量動態計算
}
寵物能量與生命值機制
聚會品質好（group_quality_score ≥ 70）：寵物獲得能量 +N，能量滿則升級。
聚會品質中等（40-69）：寵物能量不變。
聚會品質差（< 40）：寵物損失生命 -1 HP。HP 歸零後寵物進入「瀕死」狀態，下次聚會品質需 ≥ 80 才能復活。
長期未聚會（> 14 天）：能量每天自然衰減 -2，直到 HUNGRY 狀態，但不會扣 HP。

寵物頭部 = 被選中的成員的 cutout_avatar_url（去背臉部照片）
寵物身體 = 去頭的 emoji 動物圖案（成員建群時共同選擇）

3. Phase 1 · 入場定錨（NFC 主路徑 + QR fallback） v2 NFC

v2 主要入場方式改為 NFC tap；當裝置不支援或感應失敗時自動降級 QR。兩者後續流程完全一致。

3.1 NFC 可行性說明

iOS 與 Android 的 NFC 能力差異
Android：完整支援 NFC 讀寫與 P2P，所有現代裝置（Android 4.4+）內建 NFC 可用Intent-filter 自動啟動 App。
iOS：Core NFC 框架（iPhone 7+）可讀 NDEF 標籤但 P2P 已移除。iOS 11+ 需要使用者手動啟動掃描，iOS 14+ 支援背景自動偵測（需 App 註冊 domain）。
結論：NFC tag 方案可行但需確保 tag 內容為 NDEF URI 格式（https://yourapp.com/join?s=...），讓未安裝 App 的使用者也能引導到 App Store。

3.2 NFC Tag Payload 格式

// NDEF URI 格式
https://anchorapp.com/join?s={session_id}&t={short_token}

// 展開後
{
  "scheme": "anchorapp",
  "action": "join_session",
  "session_id": "uuid-here",
  "token": "6-digit-token",   // 短期憑證，伺服器驗證
  "issued_at": "ISO8601"
}

// 客戶端處理
async function handleNfcPayload(url: string) {
  const params = parseUrl(url);
  if (Date.now() - params.issued_at > 7200_000) {
    throw new Error("NFC_EXPIRED");
  }
  await joinSession(params.session_id, params.token);
}

3.3 部署策略建議

  • 場地預佈：固定聚會地點（教室、會議室）可貼永久 NFC 標籤，每次使用時主持者將 session token 寫入該 tag。
  • 主持者手機充當 tag：使用 Android HCE (Host Card Emulation) 讓主持者手機扮演 NFC tag，成員靠近即可感應。iOS 目前不支援 HCE。
  • 臨時聚會：主持者帶一張空白 NFC 標籤，App 現場寫入。或直接出 QR。

4.3 會中 Q&A 小活動控制流 v2 NEW

Q&A 是短時間（30-90 秒）的結構化互動，作為額外的定錨機制——讓群體在當下有共同聚焦的目標。主持者可在任意時間觸發，每場聚會最多觸發 3 次（避免太干擾）。

4.4 Q&A 題庫設計（可編輯） v2.1 可編輯

4.6.1 難度參數完整表

Phase 0 依情境選擇難度後，系統載入以下參數。這些參數決定聚會中的緩衝判定、品質扣分速率、事件廣播頻率。

參數 說明 L（弱） M（中） H（強）
buffer_sec_per_window 每個滾動窗口內允許的手機使用秒數 90 45 15
buffer_window_sec 滾動窗口時長 900（15 分） 900 1200（20 分）
quality_decay_rate 超額時每秒品質扣分速率 0.5 1.0 1.5
quality_recovery_rate 正常狀態每秒品質回復速率 0.3 0.2 0.1
pickup_tolerance_sec 拿起手機多久內視為「瞬間檢查」，不扣品質 15 8 3
consecutive_use_penalty_threshold 連續使用超過此秒數觸發額外扣分 180 90 45
quality_sample_interval_sec 品質取樣頻率（寫入 DB） 30 20 10
buffer_check_interval_sec 緩衝區評估循環週期 15 10 10
pin_exempt_duration_sec PIN 豁免單次時長 600（10 分） 300（5 分） 180（3 分）
pin_exempt_max_count 每成員單場豁免次數上限 3 2 1
情境疊加修正
部分情境會在難度基礎上再微調。例如 class（上課）即使選擇 H 難度，pickup_tolerance_sec 會降到 0（任何拿起即扣分）；meal（聚餐）即使選 H 難度，buffer_sec_per_window 仍維持 30 秒（留給看菜單、拍照的空間）。完整表見 §4.6.6。

4.6.3 緩衝區評估循環（每 10 秒執行）

4.6.4 品質計算公式

個人品質分數（即時）

function updatePersonalQuality(
  member: MemberLiveState,
  dt_sec: number,           // 距上次更新的秒數
  params: DifficultyParams
): number {
  let delta = 0;

  if (member.status === "EXEMPT" || member.status === "ABSENT") {
    // 不變動
    return member.personal_quality_score;
  }

  if (member.status === "NORMAL") {
    delta = +params.quality_recovery_rate * dt_sec;
  } else if (member.status === "WARN") {
    delta = 0;
  } else if (member.status === "OVER") {
    const basePenalty = params.quality_decay_rate;
    const extraPenalty = member.current_use_continuous_sec
                         > params.consecutive_use_penalty_threshold ? 2.0 : 1.0;
    delta = -basePenalty * extraPenalty * dt_sec;
  }

  const newScore = member.personal_quality_score + delta;
  return Math.max(0, Math.min(100, newScore));
}

群體品質分數（即時聚合）

function computeGroupQuality(
  members: MemberLiveState[],
  session: SessionQualityState,
  session_config: SessionConfig
): number {
  const activeMembers = members.filter(m => m.status !== "ABSENT");
  if (activeMembers.length === 0) return 0;

  // 權重 1: 成員個人品質平均（40%）
  const avgPersonalQuality = mean(activeMembers.map(m => m.personal_quality_score));

  // 權重 2: 即時專注比例（30%）
  const focusRatio = activeMembers.filter(m => m.posture === "FACE_DOWN").length
                     / activeMembers.length;
  const focusScore = focusRatio * 100;

  // 權重 3: 對話活躍度（若 VAD 啟用，20%；否則此權重歸零並提升其他）
  let voiceScore = null;
  if (session_config.vad_enabled) {
    voiceScore = session.voice_activity_ratio * 100;
  }

  // 權重 4: 階段適配度（10%）
  // 暖身期要求較低、高峰期要求高、緩和期容忍高
  const phaseModifier = {
    WARMUP: 1.1,    // 加分
    PEAK: 1.0,
    COOLDOWN: 1.05
  }[session.current_phase];

  let score;
  if (voiceScore !== null) {
    score = (avgPersonalQuality * 0.4 + focusScore * 0.3 + voiceScore * 0.2) * phaseModifier;
  } else {
    score = (avgPersonalQuality * 0.55 + focusScore * 0.35) * phaseModifier;
  }

  return Math.max(0, Math.min(100, score));
}

4.6.5 VAD 資料如何參與品質計算

VAD 啟用時，主持者裝置每秒產生一個布林樣本（是否有聲音），以及粗略的說話者切換偵測。這些資料不含任何語意內容，但能給出對話節奏的訊號。

interface VadAggregate {
  session_id: UUID;
  minute_bucket: number;                // 第幾分鐘

  voice_active_sec: number;             // 本分鐘內有聲音的秒數（0-60）
  silence_gaps: { start_sec: number, duration_sec: number }[];
  speaker_transitions: number;          // 粗略估計說話者切換次數

  // 派生特徵
  voice_activity_ratio: number;         // voice_active_sec / 60
  avg_turn_length_sec: number;          // 平均說話段長度
  participation_balance: number;        // 說話者切換頻率標準化 (0-1)
}

// 品質參與規則
function scoreConversationQuality(vad: VadAggregate): number {
  // 健康的聚會對話特徵：
  // - voice_activity_ratio 在 0.4-0.7 之間（不是獨白也不是冷場）
  // - speaker_transitions 適中（每分鐘 3-8 次，代表輪替）
  // - 不會有過長 silence_gaps（單次 > 90 秒視為冷場）

  let score = 50;  // baseline

  if (vad.voice_activity_ratio >= 0.4 && vad.voice_activity_ratio <= 0.7) {
    score += 20;
  } else if (vad.voice_activity_ratio < 0.2) {
    score -= 15;   // 過於安靜
  }

  if (vad.speaker_transitions >= 3 && vad.speaker_transitions <= 10) {
    score += 15;
  }

  const longSilence = vad.silence_gaps.filter(g => g.duration_sec > 90).length;
  score -= longSilence * 5;

  return Math.max(0, Math.min(100, score));
}
VAD 的限制
本系統 VAD 只在主持者單一裝置執行，因此無法精確辨認「誰在說話」——只能知道「有人說話 / 沒人說話」與「說話者切換的粗略次數」。這意味著 participation_balance 是近似值，不能用來評價某個成員的參與度。個人層級的參與評估得靠 Layer 2 主觀問卷。

function getPersonalizedBufferLimit(
  user_id: UUID,
  base_limit: number,
  baseline: UserBaseline | null
): number {
  if (!baseline || baseline.sample_count < 3) {
    // 資料不足，用預設
    return base_limit;
  }

  // 使用者歷史平均 vs 群體平均的 z-score
  const z = (baseline.avg_usage_per_session - GLOBAL_AVG_USAGE) / GLOBAL_STD_USAGE;

  // z > 1 （重度使用）：閾值放寬到最多 +50%
  // z < -1（輕度使用）：閾值收緊到最多 -30%
  const adjustment = Math.max(-0.3, Math.min(0.5, z * 0.2));

  return Math.round(base_limit * (1 + adjustment));
}

6.1 表現積分計算公式 v2 NEW

// 個人表現積分（每場聚會結算一次）
function computePerformanceScore(session: Session, user: User): number {
  const focusMinutes = computeFocusMinutes(session, user);           // 個人 face-down 分鐘
  const difficultyMultiplier = { L: 1.0, M: 1.3, H: 1.6 }[session.difficulty];
  const contextMultiplier = contextScoreMap[session.context] || 1.0;  // 讀書會/會議分數高
  const streakBonus = Math.min(user.current_streak * 0.05, 0.5);      // 連續日加成，上限 50%

  const basePoints = focusMinutes * 2;
  const score = basePoints * difficultyMultiplier * contextMultiplier * (1 + streakBonus);

  return Math.round(score);
}

// 每對 dyadic pair 更新（群組內）
// 例：A 與 B 本次聚會一起 → 兩人感情積分 +N
function updateRelationshipPoints(session: Session) {
  const attendees = session.attendee_uids;

  for (const [a, b] of pairs(attendees)) {
    const sessionQualityScore = computeGroupQuality(session);  // 0-100
    const relationshipDelta = Math.round(sessionQualityScore / 10);

    await db.upsert('relationship_edges', {
      user_a: a, user_b: b,
      co_session_count: increment(1),
      relationship_points: increment(relationshipDelta),
      last_met_at: session.ended_at
    });
  }
}

  let energyDelta = 0;
  let hpDelta = 0;

  if (quality_score >= 70) {
    // 好品質：餵食成功
    energyDelta = Math.round((quality_score - 50) * 0.5);  // 最高 +25
    updates.pet_status = "HAPPY";
  } else if (quality_score >= 40) {
    // 中等品質：不變
    energyDelta = 0;
    updates.pet_status = "NORMAL";
  } else {
    // 差品質：損失生命
    energyDelta = -10;
    hpDelta = -1;
    updates.pet_status = "CRITICAL";
  }

  const newEnergy = Math.max(0, Math.min(group.pet_max_energy, group.pet_energy + energyDelta));
  const newHp = Math.max(0, group.pet_hp + hpDelta);
  const newAccumulated = group.pet_accumulated_score + Math.max(0, energyDelta);
  const newLevel = computePetLevel(newAccumulated);

  updates.pet_energy = newEnergy;
  updates.pet_hp = newHp;
  updates.pet_accumulated_score = newAccumulated;
  updates.pet_level = newLevel;

  // 2. 升級時解鎖配件
  if (newLevel > group.pet_level) {
    const newUnlocks = getAccessoriesForLevels(group.pet_level + 1, newLevel);
    updates.pet_accessories = [...group.pet_accessories, ...newUnlocks];

    await sendGroupNotification(group_id, {
      type: 'PET_LEVEL_UP',
      pet_name: group.pet_name,
      from_level: group.pet_level,
      to_level: newLevel,
      unlocked: newUnlocks
    });
  }

  // 3. HP 歸零 → 瀕死警告
  if (newHp === 0 && group.pet_hp > 0) {
    await sendGroupNotification(group_id, {
      type: 'PET_CRITICAL',
      pet_name: group.pet_name,
      message: '寵物已瀕死！下次聚會品質需 ≥ 80 才能復活'
    });
  }

  await db.update('groups', group_id, updates);

  return { energyDelta, hpDelta, newLevel };
}

// 等級門檻
function computePetLevel(accumulated: number): number {
  if (accumulated < 50) return 1;
  if (accumulated < 150) return 2;
  if (accumulated < 350) return 3;
  if (accumulated < 600) return 4;
  if (accumulated < 1000) return 5;
  return 5 + Math.floor((accumulated - 1000) / 500);
}

// 長期未聚會衰減（定時任務，每天凌晨執行）
async function dailyPetDecay() {
  const groups = await db.query('groups', {
    where: { pet_last_fed_at: { lt: daysAgo(14) } }
  });

  for (const group of groups) {
    const daysSinceLastFed = daysBetween(group.pet_last_fed_at, now());
    const decayDays = daysSinceLastFed - 14;
    if (decayDays > 0) {
      const newEnergy = Math.max(0, group.pet_energy - decayDays * 2);
      await db.update('groups', group.group_id, {
        pet_energy: newEnergy,
        pet_status: newEnergy < 20 ? "HUNGRY" : "NORMAL"
      });
    }
  }
}

8. 跨階段共用邏輯

延續 v1 設計，此處僅列出 v2 調整部分。

8.1 同步與衝突

  • 兩客戶端同時回報相同 POSTURE_CHANGED：以較晚 timestamp 為準。
  • 主持者裝置斷線：推選最早 ready 的成員代為廣播 live_state。
  • NFC 感應失敗 3 次：自動切 QR UI。
  • Q&A 觸發後主持者立即結束聚會：Q&A 視為提前結束，不計入當日觸發次數上限。

8.2 推播通道（v2 擴充）

  • 即時 state：WebSocket / Firestore snapshot listener
  • Phase 4 完成通知：FCM push
  • 排行榜週更新：每週日 23:55 批次快照 + 推送周冠軍
  • 寵物狀態變化：即時推播到群組所有成員（升級、瀕死警告）

8.3 冪等性要求

所有事件 API 必須支援 idempotency_key。排行榜更新額外需要 session-level idempotency 避免重複加分：

10. 實作優先序（四 Sprint 規劃）

Sprint 1（3 週）· 最小可跑版本

  • Section 1 完整帳號系統（Email 登入即可，暫不做 OAuth）
  • Phase 0 建立 session（只支援 general 情境、難度 M）
  • Phase 1 QR 入場（NFC 暫不做）+ face-down 偵測
  • Phase 2 基本感測 + 極簡共享螢幕（計時 + 專注數）
  • Phase 3 僅支援主持者手動結束
  • Phase 4 情境層 + 連結層問卷 + 基本表現積分 + 簡版回憶錄報紙（模板化）
  • Phase 5 Job A + Job C（排行榜全域）

Sprint 2（2 週）· 群組系統

  • 完整群組 CRUD + 成員頭像去背 + 寵物人選定
  • 情境預設擴充到全部 11 種
  • Phase 5 Job B（群組卡、寵物能量/HP 初始版本）
  • Phase 5 Job D（感情積分）
  • 排行榜好友範圍

Sprint 3（2-3 週）· 進階感測與 AI

  • NFC 入場（iOS + Android 雙平台）
  • VAD 選配 + 二次同意
  • 錄音轉文字 on-device（Whisper）+ 高光時刻提取
  • AI/ML pipeline（回憶錄報紙生成、LLM 摘要）
  • 可編輯 Q&A 題庫（群組自訂題目 CRUD）
  • 個人基線 ML 開始訓練

  • 寵物升級動畫與配件系統 + 瀕死/復活流程

  • 週排行榜通知與慶祝畫面
